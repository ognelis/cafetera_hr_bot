# Оценка качества бота через RAGAS

Этот документ объясняет простыми словами, **как проект проверяет, хорошо ли бот отвечает на вопросы сотрудников**. Читать его можно без знаний про ИИ и программирование.

Официальная документация метрик (англ.): <https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/>

---

## 1. Что такое RAGAS и зачем он нужен

Наш бот работает по схеме **RAG** (Retrieval-Augmented Generation):

1. Сотрудник задаёт вопрос.
2. Система **ищет** подходящие куски из загруженных документов (регламенты, инструкции и т.п.).
3. Языковая модель **пишет ответ**, опираясь только на эти куски.

На каждом из этих шагов бот может ошибиться: найти не те документы, упустить важное, или написать то, чего в документах нет («выдумать»).

**RAGAS** — это набор автоматических «тестов», которые проверяют качество ответа и найденных документов. Вместо того чтобы человек вручную читал сотни ответов, RAGAS делает это силами ИИ и выдаёт числа от **0 до 1**, где `1` — идеально, `0` — плохо.

> Важно: RAGAS не «знает правду». Он использует ту же языковую модель, чтобы оценить сам себя. Это не идеальный, но очень полезный индикатор.

---

## 2. Как устроена проверка в проекте

Проверка состоит из двух шагов. Оба запускаются одним скриптом:

```bash
bash ragas/run.sh
```

Скрипт спросит провайдера LLM (Ollama / OpenAI / llama.cpp) и режим работы.

### Шаг 1. Генерация тестовых вопросов (`generate_testset.py`)

- Скрипт берёт все куски документов, которые уже загружены в базу знаний (Qdrant).
- Языковая модель читает эти куски и **сама сочиняет вопросы** вида «По какому документу…», «Что нужно сделать если…» и сразу записывает правильный ответ (reference).
- Результат сохраняется в `ragas/testset.json` — список пар «вопрос — эталонный ответ».

Так получается «экзаменационный билет», по которому мы будем проверять бота.

### Шаг 2. Запуск оценки (`evaluate.py`)

- Каждый вопрос из `testset.json` отправляется боту **как если бы его задал сотрудник**.
- Бот проходит весь обычный путь: ищет документы, формулирует ответ.
- Для каждого вопроса сохраняются три вещи:
  - `user_input` — сам вопрос,
  - `response` — что ответил бот,
  - `retrieved_contexts` — какие куски документов он нашёл и использовал.
- Эти три вещи передаются в RAGAS, и он считает 3–4 метрики (см. ниже).
- Итоговые оценки по каждому вопросу пишутся в `ragas/scores.json`, а в терминал выводится сводка (среднее, минимум, максимум).

## 3. Какие метрики мы считаем

Проект использует четыре метрики. Первые три — всегда, четвёртую — только если в тестовом наборе есть эталонный ответ (`reference`).

### 3.1. Faithfulness (Верность фактам) — «не выдумал ли бот»

**Что проверяет.** Все ли утверждения в ответе бота можно подтвердить найденными документами. Если бот добавил что-то «от себя» — оценка падает.

**Как считается (упрощённо).**

1. Языковая модель разбивает ответ бота на отдельные **утверждения** (простые факты).
   - Пример ответа: «Отпуск длится 28 дней, заявление подают за 2 недели».
   - Утверждения: `[«Отпуск длится 28 дней»; «Заявление подают за 2 недели»]`.
2. Для каждого утверждения модель смотрит в найденные куски документов и отвечает: «это утверждение **подтверждается** текстом или нет?».
3. Формула:

   ```
   Faithfulness = (число подтверждённых утверждений) / (общее число утверждений)
   ```

**Как читать результат.**

- `1.0` — все факты в ответе есть в документах.
- `0.5` — половина фактов выдумана или искажена.
- `0.0` — бот написал «из головы».

**На что влияет плохая оценка.** Слишком «творческий» промпт, слабая языковая модель, мало/слишком узкие документы в контексте.

---

### 3.2. Context Precision (Точность контекста) — «не нашёл ли лишнего»

В проекте используется вариант **без эталонного ответа** (`ContextPrecisionWithoutReference`).

**Что проверяет.** Из найденных кусков документов **какая доля реально помогла ответить на вопрос**. Если найдено 10 кусков, а полезны только 2 первых — точность будет невысокой.

**Как считается (упрощённо).**

1. Для каждого куска по очереди (с учётом позиции в списке) модель отвечает: «этот кусок **полезен** для ответа на вопрос или нет?» (0 или 1).
2. Применяется формула **среднее взвешенное по позиции (Average Precision)**: полезные куски в начале списка дают больше баллов, в конце — меньше.

   ```
   Precision@k = (число полезных кусков среди первых k) / k
   Context Precision = среднее Precision@k по всем полезным кускам
   ```

**Как читать результат.**

- `1.0` — каждый найденный кусок пригодился, и полезные стоят в начале.
- `0.5` — половина найденного — шум, либо полезное зарыто глубоко в списке.
- `0.0` — ничего полезного не нашлось.

**На что влияет плохая оценка.** Слабый поиск, неправильно подобранные параметры retrieval (слишком большой `top-k`, плохой реранкер), мусор в документах.

---

### 3.3. Answer Relevancy (Релевантность ответа) — «ответил ли бот на тот вопрос»

**Что проверяет.** Не «уехал» ли бот в сторону. Ответ может быть правдой, но про что-то другое — тогда он нерелевантен.

**Как считается (упрощённо).**

1. По готовому ответу языковая модель **сочиняет несколько вопросов** (обычно 3), на которые этот ответ мог бы быть ответом.
2. Каждый такой «обратный» вопрос сравнивается с исходным вопросом сотрудника через **эмбеддинги** — это способ измерить смысловую близость двух предложений числом от -1 до 1 (косинусная близость).
3. Берётся среднее значение этих близостей.

   ```
   Answer Relevancy ≈ среднее(косинус между исходным вопросом
                              и каждым сгенерированным вопросом)
   ```

**Как читать результат.**

- `1.0` — ответ строго по теме.
- `0.5` — ответ частично в сторону.
- близко к `0` — бот отвечает не о том.

**На что влияет плохая оценка.** Модель «уходит в рассуждения», промпт слишком общий, бот даёт «отказ» вместо ответа.

> Особенность: метрика чувствительна к эмбеддинг-модели. Если поменять `EMBEDDING_MODEL`, значения до и после будет честно сравнивать нельзя. Поэтому для сопоставимых прогонов фиксируйте одну и ту же модель эмбеддингов.

---

### 3.4. Context Recall (Полнота контекста) — «нашли ли всё нужное» *(опционально)*

Считается только если в тестовом наборе есть поле `reference` (эталонный ответ). В нашем проекте такие эталоны создаются на шаге генерации тестсета, поэтому метрика обычно доступна.

**Что проверяет.** Содержат ли найденные куски документов **всю** информацию, которая нужна для правильного ответа. Даже если бот отвечает верно, но поиск вытащил только половину фактов — recall будет низкий.

**Как считается (упрощённо).**

1. Эталонный ответ разбивается на отдельные утверждения.
2. Для каждого утверждения модель проверяет: «есть ли это утверждение в найденных кусках?».
3. Формула:

   ```
   Context Recall = (число утверждений из эталона,
                     нашедшихся в контексте) / (всего утверждений в эталоне)
   ```

**Как читать результат.**

- `1.0` — поиск достал все нужные факты.
- `0.5` — половина эталонных фактов не попала в контекст.
- `0.0` — нужных фактов в найденных документах нет вовсе.

**На что влияет плохая оценка.** Слишком маленький `top-k`, слишком короткие чанки, неудачная нарезка документов, плохая эмбеддинг-модель.

## 4. Пример сводки в терминале

После прогона `ragas/evaluate.py` вы увидите примерно такую таблицу:

```
============================================================
RAGAS Evaluation Summary
============================================================
  faithfulness                    avg=0.842  min=0.500  max=1.000  n=18/20
  context_precision               avg=0.763  min=0.333  max=1.000  n=20/20
  answer_relevancy                avg=0.891  min=0.642  max=0.978  n=20/20
  context_recall                  avg=0.715  min=0.250  max=1.000  n=20/20
============================================================
```

Как читать строку:

- `avg` — среднее по всем вопросам (основной показатель).
- `min` / `max` — худший и лучший случай.
- `n=18/20` — метрика успешно посчитана для 18 вопросов из 20 (по двум она не сработала, например, из-за таймаута ИИ).

Подробности по каждому вопросу лежат в `ragas/scores.json`.

## 5. Что считать «хорошим» результатом

Абсолютных порогов RAGAS не задаёт, но практические ориентиры такие:

| Метрика            | Хорошо   | Средне     | Плохо       |
| ------------------ | -------- | ---------- | ----------- |
| Faithfulness       | ≥ 0.85   | 0.70–0.85  | < 0.70      |
| Context Precision  | ≥ 0.75   | 0.55–0.75  | < 0.55      |
| Answer Relevancy   | ≥ 0.85   | 0.70–0.85  | < 0.70      |
| Context Recall     | ≥ 0.80   | 0.60–0.80  | < 0.60      |

Важнее не одно число, а **тренд**: запустили оценку до изменения и после — стало лучше или хуже.

---

## 6. Что делать при плохих цифрах

| Метрика просела    | Где искать причину                                                                 |
| ------------------ | ---------------------------------------------------------------------------------- |
| Faithfulness ↓     | Промпт разрешает «додумывать»; слабая LLM; контекст обрезается по длине.           |
| Context Precision ↓ | Слишком большой `top-k`, нет реранкера, «шумные» документы в индексе.              |
| Answer Relevancy ↓ | Бот уходит в рассуждения или отказы; промпт слишком общий; неверная эмбеддинг-модель. |
| Context Recall ↓   | Маленький `top-k`, неудачная нарезка на чанки, слабый эмбеддинг для поиска.        |

---

## 7. Требования перед запуском

Чтобы `bash ragas/run.sh` отработал, заранее нужно:

1. Загрузить документы через админ-панель — чтобы в Qdrant были куски, по которым сгенерируется тестсет.
2. Иметь рабочего LLM-провайдера (Ollama / OpenAI / llama.cpp) — он нужен и для генерации вопросов, и для оценки.
3. Запустить Qdrant (скрипт поднимает его сам через `docker compose`).

Генерация и оценка по 20 вопросам на локальной Ollama обычно занимает 10–40 минут — каждый шаг делает несколько обращений к LLM.
