## 1. Быстрый старт: готовые конфигурации

Если не хотите разбираться — просто выберите свою ситуацию:

### 🟢 Ситуация 1: «Хочу попробовать на домашнем компьютере»

**Ваше железо:** Обычный ноутбук/ПК без мощной видеокарты (или Mac M1/M2)  
**Нагрузка:** 1-2 человека используют бота  
**Качество:** Хорошее

**Что вписать в `.env`:**

```bash
LLM_PARALLEL=2
LLM_NUM_CTX=4096
LLM_MAX_TOKENS=2048
LLM_CACHE_TYPE_K=q8_0
LLM_CACHE_TYPE_V=q8_0
EMBEDDING_CHUNK_SIZE=8
DOC_QUERY_K=8
```

**Минимальные требования:** 8 GB ОЗУ (даже без видеокарты)

---

### 🟡 Ситуация 2: «Маленький офис, 5-10 сотрудников»

**Ваше железо:** ПК с видеокартой NVIDIA (RTX 4060 Ti/4070, 16 GB VRAM) или Mac M2/M3 Pro  
**Нагрузка:** 3-5 человек одновременно  
**Качество:** Отличное

**Что вписать в `.env`:**

```bash
LLM_PARALLEL=5
LLM_NUM_CTX=8192
LLM_MAX_TOKENS=4096
LLM_CACHE_TYPE_K=q8_0
LLM_CACHE_TYPE_V=q8_0
EMBEDDING_CHUNK_SIZE=8
DOC_QUERY_K=10
```

**Модель по умолчанию:** Qwen3.5-9B-Q4_K_M (~10 GB VRAM)

**Минимальные требования:** 16 GB видеопамяти (VRAM)

---

### 🔴 Ситуация 3: «Средняя компания, 20+ сотрудников»

**Ваше железо:** Сервер с мощной видеокартой (RTX 4080/4090, 16-24 GB VRAM)  
**Нагрузка:** 10-20 человек одновременно  
**Качество:** Максимальное

**Что вписать в `.env`:**

```bash
LLM_PARALLEL=10
LLM_NUM_CTX=8192
LLM_MAX_TOKENS=4096
LLM_CACHE_TYPE_K=q8_0
LLM_CACHE_TYPE_V=q8_0
EMBEDDING_CHUNK_SIZE=16
DOC_QUERY_K=15
```

**Минимальные требования:** 16-24 GB видеопамяти (VRAM)

## 2. Как узнать, сколько у вас видеопамяти

### Windows

1. Нажмите `Ctrl + Shift + Esc` → откроется Диспетчер задач
2. Перейдите на вкладку **Производительность**
3. Нажмите **GPU**
4. Найдите строку **Выделенная память GPU** (или Dedicated GPU memory)
5. Там будет число, например `12.0 GB`

**Или через командную строку:**

```cmd
wmic path win32_VideoController get name,AdapterRAM
```

---

### macOS

1. Нажмите на **яблоко** → **Об этом Mac**
2. Посмотрите строку **Графика** или **Чип**
3. Для Apple Silicon (M1/M2/M3): видеопамять общая с оперативной
   - M1/M2 с 8 GB RAM = ~6 GB доступно для GPU
   - M1/M2 Pro с 16 GB RAM = ~12 GB доступно для GPU
   - M2/M3 Max с 32+ GB RAM = ~24+ GB доступно для GPU

**Или через Терминал:**

```bash
system_profiler SPDisplaysDataType | grep "VRAM"
```

---

### Linux

```bash
# Для NVIDIA
nvidia-smi --query-gpu=name,memory.total --format=csv

# Для всех GPU
lspci | grep -i vga
```

---

### Если у вас нет видеокарты

Не проблема! Модели могут работать на обычном процессоре (CPU), но будут медленнее.

- **Без GPU:** 5-10 секунд на ответ
- **С GPU:** 1-3 секунды на ответ

Просто установите:

```bash
LLM_N_GPU_LAYERS=0  # Все вычисления на CPU
```

## 3. Основные параметры простым языком

### LLM_NUM_CTX — «Размер рабочего стола»

**Что это:** Максимальное количество токенов (слов/частей слов), которые модель может «держать в голове» одновременно.

**Аналогия:** Представьте рабочий стол. Чем он больше, тем больше документов вы можете разложить одновременно.

| Значение | Что означает | Когда использовать |
|----------|--------------|-------------------|
| 4096 | Средний стол | Простые вопросы, экономия памяти |
| 8192 | Большой стол | Стандартное значение, подходит для большинства |
| 16384 | Огромный стол | Очень длинные документы, нужно много VRAM |

**Рекомендация:** Оставьте `8192` — это золотая середина.

---

### LLM_MAX_TOKENS — «Максимальная длина ответа»

**Что это:** Сколько максимум слов может сгенерировать модель в одном ответе.

**Аналогия:** Лимит символов в SMS. Если ответ не влезает — он обрезается.

| Значение | Длина ответа | Для чего подходит |
|----------|--------------|-------------------|
| 1024 | ~700 слов | Короткие ответы, справки |
| 2048 | ~1500 слов | Стандартные HR-вопросы |
| 4096 | ~3000 слов | Развёрнутые объяснения, инструкции |
| 8192 | ~6000 слов | Очень длинные документы (редко нужно) |

**Рекомендация:** `4096` — покрывает 95% сценариев HR-бота.

---

### LLM_PARALLEL — «Количество одновременно работающих кассиров»

**Что это:** Сколько запросов может обрабатываться **одновременно**. Остальные встают в очередь.

**Аналогия:** Кассы в магазине. Если касс 5, то 5 покупателей обслуживаются сразу, остальные ждут.

| Значение | Что происходит | Когда использовать |
|----------|----------------|-------------------|
| 2 | 2 запроса сразу, остальные ждут | Домашнее использование |
| 5 | 5 запросов одновременно | Маленький офис (рекомендуется) |
| 10 | 10 запросов одновременно | Средняя компания |
| 20 | 20 запросов одновременно | Крупная компания, нужен мощный GPU |

**Важно:** Чем больше параллельных запросов, тем больше видеопамяти нужно!

**Формула:** `Видеопамять ≈ LLM_PARALLEL × 0.7 GB`

Пример: `LLM_PARALLEL=5` → нужно ~3.5 GB VRAM только на параллельные запросы.

**Рекомендация:** Начните с `5`, потом увеличивайте, если нужна большая производительность.

---

### LLM_CACHE_TYPE_K / LLM_CACHE_TYPE_V — «Сжатие рабочей памяти»

**Что это:** Квантизация (сжатие) промежуточных вычислений. Экономит видеопамять с минимальной потерей качества.

**Аналогия:** ZIP-архив для данных. Файл становится меньше, но содержимое почти не меняется.

| Значение | Экономия VRAM | Потеря качества | Когда использовать |
|----------|---------------|-----------------|-------------------|
| `f16` | 0% | 0% | Если VRAM много, нужно максимальное качество |
| `q8_0` | 50% | <1% (незаметно) | **Рекомендуется для большинства** |
| `q5_0` | 68% | 1-3% (слегка заметно) | Если VRAM мало |
| `q4_0` | 75% | 3-8% (заметно) | Только для экспериментов |

**Рекомендация:** Всегда используйте `q8_0` — экономия 50% VRAM, качество не страдает.

---

### EMBEDDING_CHUNK_SIZE — «Размер порции при индексации»

**Что это:** Сколько текстовых chunks обрабатывается за один раз при загрузке документов.

**Аналогия:** Сколько книг библиотекарь обрабатывает одновременно. Больше = быстрее, но нужно больше памяти.

| Значение | Скорость индексации | Потребление VRAM | Когда использовать |
|----------|---------------------|------------------|-------------------|
| 4 | Медленно | Мало | Ограниченная VRAM (<6 GB) |
| 8 | Средне | Средне | **Рекомендуется для большинства** |
| 16 | Быстро | Много | Много VRAM (>12 GB) |
| 32 | Очень быстро | Очень много | Только для мощных серверов |

**Рекомендация:** `8` — хороший баланс скорости и потребления памяти.

---

### DOC_QUERY_K — «Сколько документов искать»

**Что это:** Сколько фрагментов документов передаётся модели для формирования ответа.

**Аналогия:** Сколько страниц из энциклопедии вы открываете перед тем, как ответить на вопрос.

| Значение | Качество ответа | Потребление контекста | Когда использовать |
|----------|-----------------|----------------------|-------------------|
| 5 | Базовое | Мало | Простые вопросы |
| 10 | Хорошее | Средне | **Рекомендуется для большинства** |
| 15 | Отличное | Много | Сложные вопросы, много VRAM |
| 20 | Максимальное | Очень много | Только для мощных серверов |

**Рекомендация:** `10` — достаточно для 90% HR-вопросов.

## 4. Как подобрать параметры под ваше железо

### Шаг 1: Узнайте свою видеопамять

См. раздел [«Как узнать, сколько у вас видеопамяти»](#2-как-узнать-сколько-у-вас-видеопамяти).

### Шаг 2: Выберите конфигурацию

| Ваша VRAM | Рекомендуемая модель | LLM_PARALLEL | Конфигурация |
|-----------|---------------------|--------------|-------------|
| **Без GPU (только CPU)** | Qwen3.5-4B-Q4_K_M | 2 | `LLM_NUM_CTX=4096`, `LLM_MAX_TOKENS=2048` |
| **8-12 GB** | Qwen3.5-4B-Q4_K_M | 3-5 | `LLM_NUM_CTX=8192`, `LLM_MAX_TOKENS=4096`, `q8_0` |
| **12-16 GB** | Qwen3.5-9B-Q4_K_M | 3-5 | `LLM_NUM_CTX=8192`, `LLM_MAX_TOKENS=4096`, `q8_0` |
| **16-24 GB** | Qwen2.5-14B-Q4_K_M | 5-10 | `LLM_NUM_CTX=8192`, `LLM_MAX_TOKENS=4096`, `q8_0` |
| **24+ GB** | Qwen3-32B-Q4_K_M | 10-15 | `LLM_NUM_CTX=8192`, `LLM_MAX_TOKENS=8192`, `q8_0` |

### Шаг 3: Протестируйте

Запустите сервисы и проверьте потребление памяти (см. раздел [«Мониторинг»](#7-мониторинг-как-понять-что-параметры-правильные)).

**Если память заполнена на 80-90%** — всё хорошо, можно работать.  
**Если память заполнена на 95-100%** — уменьшите `LLM_PARALLEL` на 2-3.  
**Если память заполнена на 50-60%** — можно увеличить `LLM_PARALLEL` для большей производительности.

## 5. Как подобрать параметры под нагрузку

### Как оценить ожидаемую нагрузку

| Сценарий | Примерное количество RPS |
|----------|-------------------------|
| Домашнее использование | 0.1-0.5 RPS |
| Маленький офис (5 человек) | 0.5-1 RPS |
| Средний офис (20 человек) | 1-3 RPS |
| Крупная компания (100+ человек) | 3-10 RPS |

> **RPS (Requests Per Second)** — количество запросов в секунду.  
> 1 RPS = 60 запросов в минуту = 3600 запросов в час.

### Формула для подбора LLM_PARALLEL

```
LLM_PARALLEL = (Ожидаемый RPS × Средняя длина ответа в секундах) × 1.2

Примеры:
- 1 RPS × 5 секунд × 1.2 = 6 → округляем до 5
- 3 RPS × 5 секунд × 1.2 = 18 → округляем до 15
- 5 RPS × 5 секунд × 1.2 = 30 → округляем до 20 (нужен мощный GPU!)
```

### Таблица: нагрузка → параметры

| RPS | Модель | LLM_PARALLEL | Мин. VRAM | Пример GPU |
|-----|--------|--------------|-----------|------------|
| 0.5-1 | Qwen3.5-4B | 2-3 | 6-8 GB | RTX 3060, Mac M1 |
| 1-3 | Qwen3.5-9B | 3-5 | 10-12 GB | RTX 4060 Ti, Mac M2 Pro |
| 3-5 | Qwen3.5-9B | 5-8 | 12-16 GB | RTX 4070, Mac M3 Pro |
| 5-10 | Qwen2.5-14B | 8-12 | 16-24 GB | RTX 4080/4090, Mac M3 Max |
| 10+ | Qwen3-32B | 12-20 | 24+ GB | A100, H100 (серверные) |

## 6. Частые проблемы и решения

### ❌ Проблема: «Сервис не запускается, ошибка Out of Memory»

**Причина:** Нехватка видеопамяти.

**Решение:**

1. Уменьшите `LLM_PARALLEL` (например, с 10 до 5)
2. Уменьшите `LLM_NUM_CTX` (например, с 8192 до 4096)
3. Убедитесь, что `LLM_CACHE_TYPE_K=q8_0` и `LLM_CACHE_TYPE_V=q8_0`

```bash
# Попробуйте эту конфигурацию
LLM_PARALLEL=3
LLM_NUM_CTX=4096
LLM_CACHE_TYPE_K=q8_0
LLM_CACHE_TYPE_V=q8_0
```

---

### ❌ Проблема: «Ответы обрезаются на полуслове»

**Причина:** `LLM_MAX_TOKENS` слишком маленький.

**Решение:** Увеличьте `LLM_MAX_TOKENS`:

```bash
LLM_MAX_TOKENS=4096  # или даже 8192
```

---

### ❌ Проблема: «Бот отвечает медленно (10+ секунд)»

**Причина:** Модель работает на CPU вместо GPU.

**Решение:**

1. Проверьте, что видеокарта определяется:
   ```bash
   nvidia-smi  # для NVIDIA
   ```

2. Установите `LLM_N_GPU_LAYERS=99` (все слои на GPU):
   ```bash
   LLM_N_GPU_LAYERS=99
   ```

3. Если GPU нет — ничего не поделать, CPU всегда будет медленнее.

---

### ❌ Проблема: «При загрузке документов ошибка failed to find a memory slot»

**Причина:** `EMBED_CTX_SIZE` слишком маленький для текущего batch size.

**Решение:**

1. Увеличьте `EMBED_CTX_SIZE`:
   ```bash
   EMBED_CTX_SIZE=8192  # или даже 16384
   ```

2. Или уменьшите `EMBEDDING_CHUNK_SIZE`:
   ```bash
   EMBEDDING_CHUNK_SIZE=4  # вместо 8 или 16
   ```

---

### ❌ Проблема: «Качество ответов плохое, бот не находит нужную информацию»

**Причина:** Слишком мало контекста или документов для поиска.

**Решение:**

1. Увеличьте `DOC_QUERY_K`:
   ```bash
   DOC_QUERY_K=15  # вместо 10
   ```

2. Увеличьте `LLM_NUM_CTX` (если уменьшали):
   ```bash
   LLM_NUM_CTX=8192
   ```

3. Убедитесь, что документы загружены в админ-панели и обработаны.

## 7. Мониторинг: как понять, что параметры правильные

### Для NVIDIA GPU

```bash
# Откройте новый терминал и выполните:
watch -n 1 nvidia-smi
```

**Что смотреть:**

- **Memory-Usage:** Показывает использование видеопамяти
  - **50-80%** — отлично, параметры подобраны правильно
  - **80-90%** — нормально, но близко к пределу
  - **90-100%** — опасно, уменьшите `LLM_PARALLEL`
  - **<50%** — можно увеличить `LLM_PARALLEL` для большей производительности

- **GPU-Util:** Загрузка видеокарты
  - **50-90%** — отлично, GPU работает эффективно
  - **<30%** — возможно, bottleneck в CPU или памяти
  - **100%** — GPU загружен полностью, это нормально под нагрузкой

---

### Для macOS (Apple Silicon)

```bash
# Откройте новый терминал и выполните:
sudo powermetrics --samplers gpu_power -i 1000
```

**Что смотреть:**

- **GPU Power:** Потребление энергии GPU
- **Memory Pressure:** Давление на память (в Activity Monitor)
  - **Зелёный** — всё хорошо
  - **Жёлтый** — память близка к пределу
  - **Красный** — нехватка памяти, уменьшите параметры

---

### Для CPU (если нет GPU)

```bash
# Linux/macOS
top  # или htop

# Windows
# Откройте Диспетчер задач → вкладка Производительность → CPU
```

**Что смотреть:**

- **CPU Usage:** Загрузка процессора
  - **50-80%** — нормально
  - **>90%** — процессор перегружен, уменьшите `LLM_PARALLEL`

- **Memory Usage:** Использование оперативной памяти
  - Должно быть **<80%** от общей RAM

---

### Как проверить качество работы

Отправьте несколько тестовых запросов и посмотрите на время ответа:

```bash
# Тестовый запрос
curl -X POST http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [
      {"role": "user", "content": "Как оформить ежегодный отпуск по ТК РФ?"}
    ],
    "max_tokens": 500
  }'
```

**Хорошие показатели:**

- Время ответа: **1-5 секунд** (с GPU) или **5-15 секунд** (с CPU)
- Ответ полный, не обрезается
- Качество ответа высокое (информативный, точный)

**Плохие показатели:**

- Время ответа: **>20 секунд** — перегрузка
- Ответ обрезается — увеличьте `LLM_MAX_TOKENS`
- Ошибки в консоли — см. раздел [«Частые проблемы»](#6-частые-проблемы-и-решения)

## Глоссарий: что означают термины

| Термин | Простыми словами |
|--------|------------------|
| **VRAM** | Видеопамять — память видеокарты. Чем больше, тем лучше. |
| **RPS** | Requests Per Second — количество запросов в секунду. |
| **Токен** | Часть слова. 1 токен ≈ 0.75 слова на русском. |
| **Context (контекст)** | Сколько текста модель может «помнить» одновременно. |
| **KV Cache** | Временная память модели во время генерации ответа. |
| **Quantization (квантизация)** | Сжатие данных с минимальной потерей качества. |
| **GPU** | Видеокарта — ускоряет вычисления ИИ. |
| **CPU** | Процессор — работает медленнее, но тоже может запускать модели. |
| **Embedding** | Преобразование текста в числа для поиска по смыслу. |
| **RAG** | Retrieval-Augmented Generation — технология поиска + генерация ответа. |
